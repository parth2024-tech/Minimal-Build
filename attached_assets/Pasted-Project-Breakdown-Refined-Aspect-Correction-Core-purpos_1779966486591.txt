2. Revised Architecture – “Simple first, scale later”

We will not start with Kafka/ClickHouse/K8s. Instead:

```
[Client SDK] → [API Gateway (Cloudflare Workers)] → [Buffer (PostgreSQL + LISTEN/NOTIFY)]
       ↓
[Processor (Node.js or Go)] → [TimescaleDB (hypertables for time‑series)] + [Redis (real‑time cache)]
       ↓
[REST API (Next.js API routes or FastAPI)] ← [Redis] ← [Dashboard UI]
```

Why this is better for 0 → 100k events/day:

· Single Postgres + TimescaleDB extension = 10x simpler than ClickHouse. Handles 10M events/day easily with proper indexing.
· No Kafka – Use Postgres LISTEN/NOTIFY or a simple Redis stream for background processing.
· No Kubernetes – Deploy as a monolith on a $20/mo VPS (Hetzner, DigitalOcean) + Cloudflare caching.
· Scaling path – When you hit limits, you then migrate to ClickHouse. But you cross that bridge when needed.

6. Database Schema – Fixed for Deletion & Performance

```sql
CREATE TABLE events (
    id BIGSERIAL PRIMARY KEY,
    workspace_id UUID NOT NULL,
    event_name TEXT NOT NULL,
    url TEXT,
    referrer TEXT,
    ip_address INET,
    anonymized_ip INET GENERATED ALWAYS AS (set_masklen(ip_address, 24)) STORED,
    user_agent TEXT,
    properties JSONB,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
) PARTITION BY RANGE (created_at);

-- TimescaleDB turns this into a hypertable for automatic time partitioning
SELECT create_hypertable('events', 'created_at', chunk_time_interval => INTERVAL '1 day');
```

Deletion support:
DELETE FROM events WHERE workspace_id = $1 AND created_at < now() - interval '30 days' (GDPR retention).
For right to be forgotten: DELETE FROM events WHERE workspace_id = $1 AND user_identifier = $2.

7. API Endpoints – Improved

Method Endpoint Notes
POST /v1/event Idempotency via client_event_id header (store hash in a separate processed_events table for 48h).
GET /v1/query/timeseries Returns pre‑aggregated (daily/hourly) from a materialized view refreshed every 5 minutes – not real‑time.
GET /v1/live WebSocket that subscribes to Redis counters only for the last 5 minutes.

Why materialized views? Raw queries on events table become slow. Instead, create a continuous aggregate (TimescaleDB):

```sql
CREATE MATERIALIZED VIEW daily_metrics
WITH (timescaledb.continuous) AS
SELECT workspace_id, event_name, time_bucket('1 day', created_at) AS day, count(*)
FROM events GROUP BY 1,2,3;
```

Dashboards query this – sub‑second.

8. Real‑Time WebSocket – Properly Engineered

· Not for all data – only for “live” view (last 5 minutes, aggregated every 10 seconds).
· Backend: Redis pub/sub per workspace. Each dashboard subscribes to live:workspace:{id}.
· Connection limit: 5k concurrent per server (Node.js with ws library). For more, use Pusher/Ably (SaaS) instead of self‑hosting.
· Scaling: When >5k connections, deploy separate WebSocket server (Dedicated on Fly.io).

9. Export System – Detailed

Sync export (fast, limited):
GET /v1/export?format=csv&limit=10000 → returns CSV stream directly.

Async export (large):

1. POST /v1/export/job → returns job_id and job_poll_url.
2. BullMQ worker streams data from Postgres cursor → writes to S3 (or local disk) in 10MB chunks → concatenates.
3. User polls GET /v1/export/job/{id} → when status=complete, get presigned URL valid for 1 hour.
4. Email notification (if user opted in).

Cost: S3 for exports: $0.023/GB. 10k exports of 10MB each = $2.30 – fine.

10. Security & Privacy – Fixed

Issue Solution
GDPR right to deletion DELETE endpoint + automated weekly job to hard‑delete anonymized old data.
IP anonymization Store only /24 mask (last octet zeroed). Geolocation from anonymized IP is safe.
Cookie consent Use localStorage for anonymous ID, not cookies. For cookie banner, integrate with react-cookie-consent.
Data retention Configurable per workspace (default 30 days raw, 12 months aggregated).
API key rotation Provide UI to regenerate; old keys invalidated after 24h grace period.
