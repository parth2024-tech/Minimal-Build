Why move from TimescaleDB?

· TimescaleDB (Postgres) works well up to ~10M events/day with proper indexing, but:
  · GROUP BY queries on high‑cardinality columns (e.g., url) become slow.
  · Storage grows faster (row‑oriented).
  · No native sharding – you’d need Citus, which adds complexity.
· ClickHouse is columnar, compresses data 5‑10x, and returns aggregates on billions of rows in milliseconds.

Migration strategy – zero downtime

We will dual‑write for a transition period:

```
[Client SDK] → [Ingestion API] → [Write to both TimescaleDB (old) + ClickHouse (new)]
                              ↓
                    [Dashboard reads from ClickHouse (new) after validation]
```

Step‑by‑step:

1. Provision ClickHouse cluster (start with a single node, later sharded). Use ClickHouse Cloud (starts at $0.12/hr) or self‑host on a Hetzner CX42 ($40/mo).
2. Create the events table in ClickHouse with the same schema (but adapted for ClickHouse).
3. Backfill historical data from TimescaleDB to ClickHouse using clickhouse‑client with parallel streaming.
4. Modify ingestion service to write to both databases asynchronously (fire‑and‑forget to ClickHouse, with a retry queue).
5. Switch dashboard queries to read from ClickHouse via a new query service.
6. Run dual‑read comparison for 1 week – compare results, log discrepancies.
7. Disable writes to TimescaleDB after confidence is high, keep it as a cold backup.
8. Decommission TimescaleDB (or keep for metadata).

ClickHouse Schema (Optimized)

```sql
CREATE TABLE events (
    event_id UUID,
    workspace_id UInt64,
    event_name LowCardinality(String),
    url String CODEC(ZSTD(3)),
    referrer String CODEC(ZSTD(3)),
    country_code FixedString(2),
    device_type LowCardinality(String),
    anonymized_ip IPv4,
    properties JSON,
    timestamp DateTime64(3, 'UTC')
) ENGINE = MergeTree
PARTITION BY toYYYYMM(timestamp)
ORDER BY (workspace_id, event_name, timestamp)
SETTINGS index_granularity = 8192;
```

Key optimizations:

· LowCardinality for enums – reduces storage.
· CODEC(ZSTD) for high‑entropy strings.
· Partition by month (not day) – avoids too many parts.
· Order by (workspace_id, event_name, timestamp) – most queries filter by workspace + event + time.

2. Advanced Segment Engine

Problem with naïve segments

If a user defines a segment url LIKE '%pricing%' AND country_code = 'US', querying raw events every time is expensive.

Solution: Pre‑computed segment rollups

Approach: Use ClickHouse materialized views that pre‑aggregate common segment dimensions.

Create a table for segment snapshots:

```sql
CREATE TABLE segment_daily_metrics (
    workspace_id UInt64,
    segment_id UInt64,
    event_name LowCardinality(String),
    date Date,
    count UInt64,
    unique_users UInt64  -- using uniqCombined
) ENGINE = SummingMergeTree
ORDER BY (workspace_id, segment_id, date);
```

Then, for every saved segment, create a materialized view that filters events and populates this table incrementally.

But you cannot create a MV per user segment dynamically (too many MVs). Instead, use bitmap indexes or bloom filters for ad‑hoc segmentation. The industry standard:

· For saved segments (up to 100 per workspace), use a separate aggregation table that is refreshed every hour via a scheduled job.
· For ad‑hoc filters (user drags a slider), query the raw events table but with sampling – ClickHouse has SAMPLE clause.

Implementation (production)

```sql
-- Segment definition stored in PostgreSQL (metadata)
CREATE TABLE segments (
    id SERIAL PRIMARY KEY,
    workspace_id UUID NOT NULL,
    name TEXT,
    conditions JSONB  -- e.g., [{"field":"url","op":"contains","value":"/pricing"}]
);

-- Hourly job (cron) to refresh segment aggregates for the last 24h
-- In Go/Python: for each segment, generate ClickHouse SQL
INSERT INTO segment_daily_metrics
SELECT 
    workspace_id,
    {segment_id} as segment_id,
    event_name,
    toDate(timestamp) as date,
    count() as count,
    uniqCombined(user_id) as unique_users
FROM events
WHERE workspace_id = {workspace_id}
  AND timestamp >= now() - INTERVAL 1 DAY
  AND {condition_sql}   -- e.g., url LIKE '%/pricing%'
GROUP BY workspace_id, event_name, date
```

Performance: 100 segments × 1 day of data = ~100 queries. Each query runs in <2 seconds on 100M rows. Schedule this hourly, outside peak hours.

3. Real‑Time Dashboard at Scale

Problem with WebSocket + Redis pub/sub

When you have 5,000 dashboards each subscribing to a workspace channel, Redis pub/sub struggles with fan‑out. Also, reconnection storms after a deployment can kill the server.

Improved architecture

We move from broadcast to pull‑based with long polling for most users, and dedicated WebSocket only for paying customers who need live view.

New design:

```
[Client] → (optional: WebSocket) → [Live Aggregator Service] → [Redis Streams]
                      ↑
                      └── (fallback: Server‑Sent Events or polling every 5s)
```

Live Aggregator Service (Go or Rust):

· Maintains a sliding window of the last 5 minutes of events per workspace.
· Uses Redis Streams (XADD, XREADGROUP) to consume events from the ingestion pipeline.
· Computes counters incrementally (in‑memory map with expiry).
· Exposes a gRPC endpoint for the dashboard backend to fetch live counts.

Why this scales: No per‑client WebSocket connection to ClickHouse. The aggregator is stateless and can be horizontally scaled using consistent hashing on workspace_id.

Implementation steps for Phase 2

1. Build live aggregator (Go):
   · Subscribe to Redis Streams (ingestion writes a copy to stream:live).
   · Maintain sync.Map of workspace_id → LiveCounts (reset every minute).
   · Expose HTTP endpoint GET /live/:workspace_id that returns JSON.
2. Modify dashboard to poll this endpoint every 2 seconds (instead of WebSocket).
3. Add WebSocket only for enterprise – the aggregator can also handle WebSocket connections using gorilla/websocket with a connection pool (max 10k per instance).

Cost: A single c5.large (2 vCPU, 4GB) handles 50k req/sec of polling + 10k WebSockets.

4. Multi‑Region Deployment for Latency

Problem

If your users are in EU and US, querying a single ClickHouse cluster in us‑east‑1 adds 100ms latency.

Solution

Deploy active‑active ClickHouse with replication across regions. But that’s expensive and complex. Instead:

· Read replicas per region: ClickHouse supports remote table function. Create a cluster with one node per region.
· Geo‑routing via Cloudflare Workers: Route API requests to the nearest region based on Cf-Ray header.

Simpler alternative (for Phase 2): Keep single region, but use Cloudflare CDN for dashboard assets and query caching. Most queries are repeatable (e.g., “last 7 days” for a dashboard). Cache them for 60 seconds at edge.

6. Performance Tuning (ClickHouse)

Here are the exact settings you need to apply for production:

```xml
<!-- config.xml -->
<merge_tree>
    <max_parts_in_total>100000</max_parts_in_total>
    <parts_to_delay_insert>1500</parts_to_delay_insert>
    <parts_to_throw_insert>3000</parts_to_throw_insert>
</merge_tree>

<dictionaries_config>*_dictionary.xml</dictionaries_config>
<compression>
    <case>
        <method>zstd</method>
        <level>3</level>
    </case>
</compression>
```

Key tuning steps:

1. Disable fsync for inserts if you can tolerate <1 second data loss (set fsync_metadata = false). Improves insert throughput by 10x.
2. Use async_insert – ClickHouse buffers inserts and writes in batches. Enable via async_insert=1&wait_for_async_insert=0.
3. Materialized views for common queries:

```sql
CREATE MATERIALIZED VIEW events_hourly_mv
ENGINE = SummingMergeTree ORDER BY (workspace_id, hour)
AS SELECT
    workspace_id,
    toStartOfHour(timestamp) as hour,
    event_name,
    count() as total_events
FROM events GROUP BY 1,2,3;
```

Dashboard queries hour instead of raw events.

11. Potential Pitfalls & Mitigations (Phase 2)

Pitfall Mitigation
ClickHouse insert bottlenecks Use async_insert + batch size of 100k rows. Monitor too many parts error – increase max_parts_in_total.
Segment refresh job takes too long Incremental refresh (only last 24h). If still slow, move to triggers (materialized views per segment – but beware of MV explosion). Tradeoff: Accept 1 hour staleness for custom segments.
Live aggregator memory blow Use TTL on in‑memory maps (delete workspaces inactive > 5 min). Limit to top 1000 workspaces; others fall back to polling ClickHouse directly.
Dual‑write inconsistency Implement a reconciliation job that runs nightly, compares counts, and backfills missing events.
Cost overrun Set budget alerts in your cloud provider. Use ClickHouse Cloud’s autosuspend for idle clusters (not needed for production).
